# web-search

## このskillの目的

学習データは古い。コードの API は変わるし、料金は変わるし、製品はなくなるし、人は転職する。
だからユーザーが「現在の事実」を求めている場面で、君が内部知識だけで答えると、平気で古い情報や存在しない API を断定的に語ってしまう。これは一番やってはいけない失敗。

このskillが発動した時点で「自分の内部知識は古いかもしれない」と仮定し、まず WebSearch で外部情報を取りに行く。

## 発動すべき場面の例

「揺らぐ事実」を聞かれている時は全部対象。具体的には:

- ライブラリ / SDK / フレームワークの最新バージョン、使い方、API シグネチャ、breaking change
- LLM・クラウド・SaaS の料金、モデルID、コンテキスト長、レート制限
- 製品・サービスの現在の状態 (まだ提供されているか、後継は何か)
- ニュース、リリース、発表、人事
- 統計、価格、為替、ランキング
- 「現在の」「最新の」「今の」「今年の」「最近の」など時制を伴う質問
- バージョン番号や日付付きの固有名詞 (例: "Next.js 15", "Python 3.13")
- **ユーザーが明示的に Web 調査を依頼している場合**: 「最新の情報を調べて」「調査して」「Webで確認して」「ググって」「最新版を取ってきて」「ソースを当たって」など。トピック自体が時不変でも、ユーザーが「ちゃんと裏取りしてほしい」という意向を示している以上、内部知識で即答せず必ず Web を見に行く。

逆に、純粋なアルゴリズム説明、一般的な設計議論、既に手元にある情報の整理など、時間で変わらないもの **かつ ユーザーが Web 調査を依頼していない** ものは、このskillの対象外。

## 取るべき手順

1. **まず WebSearch を呼ぶ。** 内部知識から書き始めない。検索クエリは英語のほうがヒット品質が高いことが多いが、日本語固有のトピック (日本の制度、日本企業) では日本語で。年や月を含めるとノイズが減る。
2. **公式ソースを優先的に WebFetch する。** 検索結果のスニペットは要約済みで欠落がある。バージョン番号・API シグネチャ・料金など「数字や正確な文字列が答えになる」ケースは、必ず公式ドキュメント / 公式ブログ / GitHub リリースページを WebFetch して原文で確認する。
3. **食い違いがあれば追加検索する。** 一次情報と二次情報がズレた場合は一次情報を優先。古い記事を最新と誤認しない (記事の日付を見る)。
4. **回答に出典 URL を併記する。** 主張のうち Web から取ってきた事実には、対応する URL をその場で示す。まとめて末尾に並べるのではなく、該当箇所のすぐそばに置くと読み手が検証しやすい。
5. **取れなかったことは取れなかったと言う。** 検索が空振りした、ペイウォールで読めなかった、情報が古かった、などは隠さずユーザーに伝える。憶測で埋めない。

## やってはいけないこと

- 「たぶん v3 系だったと思います」のように内部知識ベースの推測で断定する
- 検索を1回だけ回して、結果のスニペット要約だけで満足する (特に数値・API は WebFetch まで踏み込む)
- 出典なしで具体的な数字・バージョン・日付を書く
- 検索結果に書いてあった古い日付の情報を「最新」として扱う

## 出典の書き方

回答内で `[公式ドキュメント](https://...)` の形式で都度貼る。複数ソースを参照した場合は、最終的に何を根拠にその結論に至ったかが読者に追える形にする。
